## 1. Goal Statement

Ship **`io.github.kotindia:core:0.2.0`** to Maven Central — an evolved core library that takes KotIndia from a *validation library* to a *complete validation toolkit usable in real production apps on every platform Kotlin runs on.*

The thesis: `core:0.1.0` answered "is this Aadhaar valid?" `core:0.2.0` answers everything around that — incremental input, typed error context, multi-platform reach, more validators, and a faster runtime — without forcing UI or copy decisions on consumers.

**Explicitly NOT in scope (rejected during brainstorming):**

- A `:compose` module of any kind (opinionated TextFields OR headless state holders).
- Localized strings in any language. The library ships zero copy.
- Any UI framework dependency.

## 3. Non-Goals (rejected — do not build)

| # | Non-goal | Why rejected |
|---|---|---|
| NG1 | A `:compose` module (opinionated TextFields like `AadhaarTextField`) | Forces 8+ design decisions onto every consumer (Material variant, theming, icons, copy, layout, animation, spacing). Almost no team adopts opinionated UI from a third-party validator library. Comparable category libraries (TanStack Table, Reka UI, Headless UI) all converged on headless. |
| NG2 | Headless `:compose-state` module (`rememberAadhaarFieldState()`) | Borderline justified — consumers can write the state holder themselves in 30 lines from a recipe. A separate module adds version coupling to Compose Multiplatform without much code-saving for users. Revisit only if GitHub demand emerges. |
| NG3 | Localized error strings in the library (English / Hindi / regional languages) | Library would fight every consumer's i18n pipeline (Crowdin, Phrase, in-house resource files). Typed `ValidationContext` lets consumers render their own copy in any language with full parameter access. Zero strings = zero conflicts. |
| NG4 | Speculative validators beyond the chosen 3 | Building speculatively is an OSS trap. v0.1.0 hasn't even publicly launched. Three is enough to demonstrate the pattern; v0.2.1+ absorbs real GitHub demand reactively. |
| NG5 | Linux-native KMP target | JS + WASM cover server-side and browser runtimes. Native Linux is rare for validation use cases. Defer to v0.3.0 if a Ktor-on-Linux contributor opens a PR. |
| NG6 | Async / coroutine-based validators | All validation is pure-function and synchronous. Adding `kotlinx.coroutines` dependency violates the zero-runtime-dep guarantee. |
| NG7 | KSP / compiler plugin to generate validators from annotations | Annotation processors add build-time complexity. Manual implementation is a few hundred lines per validator and produces clearer KDoc. |

## 4. Hard Guardrails (carried over + new)

### Carried from `core:0.1.0`

All v0.1.0 guardrails remain in force (see `docs/PROJECT_PLAN.md §3`). Specifically the merge-blockers:

- Zero runtime deps in `core` (`enforceZeroDeps`)
- `explicitApi()` STRICT
- 100% line coverage via Kover
- Binary-compatibility-validator gate
- All public APIs documented (KDoc + `@sample`)
- Conventional commits, one-slice-per-PR
- `kotest-runner-junit5` permanently banned

### New for v0.2.0

| Guardrail | Enforcement |
|---|---|
| `ValidationContext` is sealed and typed — no `Map<String, Any?>` or stringly-typed parameters | Compiler — sealed class with exhaustive `when` |
| Every `Invalid` result MUST carry a `ValidationContext` (even `ValidationContext.None`) — never null | Compiler signature — `data class Invalid(val reason: InvalidReason, val context: ValidationContext)` |
| `validateProgressive` MUST return `ProgressiveResult.Typing` for every input length 1..maxLength-1 of the right character class — never `Invalid` mid-typing | Anika test gate per validator |
| Regex objects MUST be top-level `private val` constants in each validator file — not constructed inside `validate()` | Marcus arch review; PR check |
| JS target MUST not introduce `kotlin-stdlib-js` browser-specific APIs in `commonMain` — `js("IR")` only, no JS interop in shared code | Marcus arch review |
| WASM target MUST publish only `wasm-js` (not `wasm-wasi`) for v0.2.0 — Compose Multiplatform's WASM target | Karan build config |
| New validators (DIN, Voter ID, FSSAI) MUST follow the slice template — single PR each, format-only is fine | Vaishali PRD authoring |
| `docs/RECIPES.md` snippets MUST be copy-pasted into a new project and verified to compile before merge | Anika gate |
| Release `core:0.2.0` only after Sandeep explicit go-ahead | Same as v0.1.0 — no autonomous publish |

### Backward compatibility

This is the most important guardrail. **`core:0.1.0` consumers MUST be able to upgrade to 0.2.0 without code changes.**

- `validate()` keeps its current signature (`fun validate(value: String): ValidationResult`).
- `Invalid(reason)` v0.1.0 callers must still work. Mechanism: add a secondary constructor `Invalid(reason: InvalidReason)` that defaults `context = ValidationContext.None`. Or change the data class to `Invalid(reason, context = ValidationContext.None)` so existing callers don't need to pass context.
- `binary-compat-validator` will block the PR if any 0.1.0 public symbol's signature changes incompatibly.
- All new APIs are pure additions (`validateProgressive`, `validateAndFormat`, `sanitize`, etc.).

## 5. Architecture

### 5.1 Public API additions

#### State machine for incremental input

```kotlin
public sealed class ProgressiveResult {
    public data object Empty : ProgressiveResult()
    public data class Typing(val visualText: String) : ProgressiveResult()
    public data object Valid : ProgressiveResult()
    public data class Invalid(
        val reason: InvalidReason,
        val context: ValidationContext,
    ) : ProgressiveResult()
}

public object Aadhaar {
    public val maxLength: Int get() = 12
    public val allowedChars: CharRange get() = '0'..'9'

    public fun sanitize(rawInput: String): String  // strips non-allowed, ASCII-only, caps at maxLength
    public fun validateProgressive(value: String): ProgressiveResult
    public fun validateAndFormat(value: String): ValidatedFormat
    // existing v0.1.0 API unchanged
}
```

#### Typed validation context (replaces stringly-typed parameters)

```kotlin
public sealed class ValidationContext {
    public data object None : ValidationContext()
    public data class LengthMismatch(val expected: Int, val actual: Int) : ValidationContext()
    public data class CharacterAt(val position: Int, val char: Char) : ValidationContext()
    public data class ChecksumMismatch(val computed: Char, val provided: Char) : ValidationContext()
    public data class PrefixMismatch(val expected: Set<Char>, val actual: Char) : ValidationContext()
    public data class CategoryMismatch(val validCategories: Set<Char>, val actual: Char) : ValidationContext()
}

public sealed class ValidationResult {
    public data object Valid : ValidationResult()
    public data class Invalid(
        val reason: InvalidReason,
        val context: ValidationContext = ValidationContext.None,
    ) : ValidationResult()
}
```

#### Combined validate-and-format

```kotlin
public data class ValidatedFormat(
    val raw: String,
    val formatted: String,
    val masked: String?,
    val result: ValidationResult,
)
```

### 5.3 New validators (each = own slice)

| Validator | Format | Checksum | Slice |
|---|---|---|---|
| DIN (Director Identification Number) | `\d{8}` — 8 digits, no internal structure | None — MCA-issued without public algorithm | 20a |
| Voter ID / EPIC | `[A-Z]{3}\d{7}` — 3 letters + 7 digits, state-prefix varies | None — format-only | 20b |
| FSSAI License | `\d{14}` — 14 digits, structured (1-digit license type + 2-digit state + 2-digit year + 2-digit registering authority + 7-digit assigned number) | None — format + structural validation only | 20c |

Each follows the existing `core` per-validator template (`commonMain` validator file + `commonTest` test file + KDoc with `@sample` + ReferenceVectorsTest).

### 5.4 Performance: regex caching

```kotlin
// File: core/src/commonMain/kotlin/io/github/kotindia/Aadhaar.kt
private val AADHAAR_REGEX: Regex = Regex("^\\d{12}$")  // top-level, compiled once

public object Aadhaar {
    public fun validate(value: String): ValidationResult {
        // ...
        if (!AADHAAR_REGEX.matches(value)) return Invalid(...)
        // ...
    }
}
```

Pattern applied to all 19 validators. Allocation reduction is automatic — `private val` constants are class-init-time, JIT-friendly.

### 5.5 JS / WASM targets

Add to `core/build.gradle.kts`:

```kotlin
kotlin {
    js(IR) {
        nodejs()
        browser()
    }
    @OptIn(ExperimentalWasmDsl::class)
    wasmJs {
        nodejs()
        browser()
    }
    // existing targets unchanged
}
```

No commonMain changes needed — all current code is pure Kotlin stdlib, K/JS and K/WASM compile it as-is.

Tests run on Node.js for both targets via the standard Karma/Mocha stack provided by KGP.

### 5.6 `docs/RECIPES.md`

Single markdown file, ~6 recipes:

1. Compose Multiplatform `TextField` with progressive validation (Android + Desktop + iOS)
2. Compose with custom design system / `BasicTextField`
3. Spring Boot `@Validated` integration

5. Node.js / Express server using the JS artifact

Each recipe is a **complete, copy-paste-able snippet** verified to compile. The file is in the repo, not the published library — but it's referenced from the README so it shows up on Maven Central's project landing page.

## 8. Backward Compatibility

This is non-negotiable. `core:0.1.0` consumers upgrading to `0.2.0` MUST compile without changes.

### Mechanism

1. **`Invalid` data class** gets a `context = ValidationContext.None` default parameter:

   ```kotlin
   // v0.1.0
   public data class Invalid(val reason: InvalidReason) : ValidationResult()

   // v0.2.0
   public data class Invalid(
       val reason: InvalidReason,
       val context: ValidationContext = ValidationContext.None,
   ) : ValidationResult()
   ```

   Existing callers `Invalid(InvalidReason.WRONG_LENGTH)` continue to work. Pattern-matching on `is Invalid` continues to work. Destructuring `(reason)` continues to compile (data class drops the second component for compatibility).

2. **`InvalidReason` enum** is **not** changed. No new entries. Adding entries would be a binary-compatibility break.

3. **All v0.1.0 validators** keep their existing public API. New methods (`validateProgressive`, `sanitize`, etc.) are pure additions.

4. **Binary-compatibility-validator** runs on every PR. The 0.1.0 API dump (`core/api/core.api`) is the baseline; 0.2.0 PRs may only add to it.

### Risk

The data class secondary constructor for `Invalid(reason)` may not actually preserve binary compatibility with apiCheck — the data class gets a synthetic `copy(reason, context)` method that didn't exist in 0.1.0. **Marcus to verify in slice 15.** If apiCheck flags this, fallback is to make `ValidationContext` a separate field on the result returned by a new method `validateWithContext()`, leaving v0.1.0's `Invalid(reason)` shape untouched.

## 12. Open Questions for PRD authoring

These are deliberately deferred to slice-level PRDs (Vaishali) — not blockers for this design.

1. **`ValidationContext` for which `InvalidReason` entries?** Some reasons (`EMPTY`) need no context; others (`WRONG_LENGTH`, `INVALID_CHECKSUM`, `INVALID_PREFIX`, `INVALID_CATEGORY`) clearly do. Slice 15 PRD enumerates which goes with which.

2. **`validateProgressive.Typing` payload — what's in `visualText`?** Probably the current `format()` output for partial input (e.g., "1234 567" while typing 7-of-12-digit Aadhaar). Slice 14 PRD specifies the exact transformation.

3. **`validateAndFormat` for non-formattable validators?** Some validators (Mobile, Pincode) may not have a meaningful `formatted` form distinct from raw. Slice 16 PRD specifies whether `formatted == raw` or whether the method is omitted on those validators.

4. **Performance benchmark methodology in slice 17.** What tool — `kotlinx.benchmark`? Plain JVM microbenchmark? Slice 17 PRD picks.

5. **`RECIPES.md` Spring Boot / Ktor recipe scope.** Are we writing a complete REST endpoint or just the validator wiring? Slice 21 PRD scopes.
